# ADR-0011 · Provably-fair derivation and game primitives

**Status:** accepted · 2026-04-19

## Context

ADR-0004 pinned commit-reveal as the RNG story but stopped at "HMAC-SHA256 of server seed over `{client_seed}:{nonce}`"; the actual byte-to-outcome mapping and the cursor semantics for games that need multiple draws per round (Mines' tile permutation, Plinko's per-peg bounces) were unspecified. At the same time, Phase 7 introduced four destinations (`/casino/crash`, `/casino/mines`, `/casino/plinko`, `/casino/lottery`), and three of them will ship "coming soon" until their logic phase lands — but every one still needs the same chrome: a bet input, a post-round banner, a provably-fair receipt, and a shell that places them. Without a shared primitives layer and a single route contract, each game would grow its own copy and drift.

## Decision

- **Byte derivation is HMAC-SHA256 per RFC 2104, advanced by a cursor.** `deriveBytes(serverSeed, clientSeed, nonce, cursor=0, count=32)` computes `HMAC_SHA256(serverSeed, ` \`${clientSeed}:${nonce}:${cursor}` )` — the cursor is an integer that increments after each 32-byte digest is exhausted, so a single round can consume hundreds of bytes (Plinko 16-row, Mines 25-tile permutation) without collision. `deriveFloats` decodes four bytes per float via the positional sum `Σ byte[i] / 256^(i+1)` to yield a uniform `[0, 1)` value. The module imports no DB, WS, or framework code; its only dependency is `node:crypto`. Determinism is covered by 12 unit tests including a 10,000-run identity check and a 1M-float distribution check across ten buckets.
- **Two component locations for game code, not one.** `src/components/game/` holds cross-cutting primitives (`BetControls`, `WinReveal`, `ProvablyFairBadge`, `GameShell`, `GamePlaceholder`); `src/components/originals/<slug>/` (future) holds per-game composition (board renderers, animations, per-game controls). Primitives take data, emit intents, and know nothing about the game they're placed in — the same `<BetControls stake onStakeChange onPlaceBet onCashOut phase>` drives Crash, Mines, and Plinko unchanged. `GameShell` owns the breadcrumb + header + two-column grid; the stage pane is a slot the game fills.
- **One dynamic route for every game.** `src/app/(app)/casino/[game]/page.tsx` validates `params.game` against `resolveGames()` and calls `notFound()` when the slug is unknown or `enabled === false`. `generateStaticParams` pre-renders the enabled set, so flipping `FLAG_MINES=off` removes `/casino/mines` from the build without a code change. The page reads the session once via `requireSessionUser({ redirectTo })` (a thin wrapper over `auth()` + user row that replaces the two places this pattern was duplicated), then renders `GamePlaceholder`. When a real game ships, its page replaces the placeholder render with an originals-composed stage — the shell, controls, and route remain.
- **`/dev/*` is gated by `NODE_ENV`, not by an env flag.** The existing `/dev/tokens` playground gained a sibling `/dev/primitives` sandbox that renders each primitive in every state against fixture seeds. Both sit under `src/app/dev/layout.tsx`, which calls `notFound()` when `NODE_ENV === "production"` — one file closes the whole tree in prod while leaving local `next dev` untouched. A Playwright spec runs the prod build and asserts 404.

## Consequences

- **Positive.** Every game's page is `<GameShell title tag status stage side />` — adding Lottery's UI is composing a `<LotteryStage>` and handing it to the shell, not rebuilding the chrome. Coming-soon pages are real routes that exercise every primitive except the stage, so the visual cost of the unbuilt games is near zero. Seed reveal is reproducible from storage alone (`server_seed`, `client_seed`, `nonce`, cursor=0 for single-draw games) — the verifier page in a later phase is ten lines of glue. `NODE_ENV` gating means we can't accidentally ship a surface without flipping an env var, and the e2e spec makes the assertion a CI failure, not a convention.
- **Negative.** Cursor discipline is now load-bearing: a game that miscounts bytes (reads the same cursor twice, forgets to advance) will silently produce correlated outcomes that still pass the "is it HMAC-SHA256" audit. Mitigation is keeping the cursor in the derivation module and handing each game a typed iterator (`nextByte`, `nextFloat`, `nextInt(max)`) — not raw offsets. The two-location split (`game/` vs `originals/<slug>/`) is a convention reviewers must enforce in PRs; there's no tooling that fails a build when a primitive leaks game-specific imports.
- **Reversible.** Swapping HMAC-SHA256 for a VRF or on-chain beacon (ADR-0004 already flagged this) is still a module-level change. Splitting the `[game]` segment back into per-game pages is mechanical if we ever need game-specific metadata that outgrows `resolveGames()`. The `/dev/*` gate is one `if` away from being a per-route check if some dev surface ever needs to ship to production.

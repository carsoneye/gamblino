# ADR-0010 · Route groups, nav model, and shell separation

**Status:** accepted · 2026-04-19

## Context

ADR-0009 pinned AppShell topology during Phase 6 scaffolding. An end-to-end audit after the first shell commit surfaced a cluster of related problems: the marketing `/` and authed `/casino` routes shared one `<AppShell>`, so the logged-out landing rendered a disabled sidebar and chat rail (dead chrome); `src/lib/nav.ts` hardcoded every game to `status: "coming-soon"` regardless of `env.FLAG_*`; the lobby grid lied about interactivity (coming-soon cards rendered as `<div>` but kept `hover:border-accent`); the `game-art.tsx` file grew to 537 lines with 7 colliding `@keyframes` and a parallel OKLCH palette that drifted from the `@theme` tokens; the signin action hardcoded `redirect("/casino")` and ignored `callbackUrl`; the ChatRail leaked the phase plan ("lights up in phase 7"); and the sidebar had no responsive drawer below `md`. These are not separate issues — they share a root cause in how Phase 6 bundled "shell + landing + lobby" into one monolith without giving the three surfaces distinct contracts.

## Decision

- **Route groups, not branching in a single shell.** `(marketing)` wraps `/` in a thin `MarketingShell` (sticky header, hero-first main, footer). `(app)` wraps `/casino` (and future authed routes) in the auth-guarded `AppShell` from ADR-0009. `(auth)` keeps its centered narrow shell. The auth check moves to `(app)/layout.tsx` as a single `auth()` + user row read, passed down to `AppShell` — pages no longer duplicate the check.
- **Nav catalog becomes a three-part export.** `marketingUtility` feeds the MarketingShell's in-page anchor nav. `userMeta` lists authed user routes with explicit `enabled` flags (Leaderboard / Transactions / Profile land in Phase 14 — they render as disabled until then, not as broken links). `gameTaxonomy` + `resolveGames()` derives each game's `status` from the `FLAG_*` env var at render time, restoring the kill-switch-without-redeploy story that ADR-0009 promised but the hardcoded `status` field silently broke.
- **Truth in affordance for game cards and nav items.** Coming-soon game cards render as `<div aria-disabled="true">` with no hover-accent border and no pointer cursor; live cards render as `<Link>` with a real focus ring. Sidebar game entries do the same — `<Link>` when live, `<div>` when pending. No component styles hover affordances it can't fulfill.
- **Shell components take one user prop, not an overloaded union.** TopBar is `{ user?: User }` instead of the `AuthedProps | Record<string, never>` discriminator. Lowercase `credits` label (the all-caps letter-spaced micro-chip is an anti-pattern called out in the project's design canon).
- **Mobile drawer via Base UI Dialog.** Below `md`, the static sidebar is hidden and a hamburger in the TopBar opens a Base UI `Dialog` positioned as a left sheet. The same `<Sidebar variant="drawer">` renders inside — one source of nav items for both breakpoints. Closes on route change via `usePathname()`.
- **Game art split into one file per game + shared tokens.** The 537-line monolith becomes `game-art/{index,crash,mines,plinko,lottery,tokens}.tsx`. Each art file uses `var(--color-accent)` etc. instead of inline OKLCH literals. Keyframes are gated behind `@media (prefers-reduced-motion: no-preference)` so motion turns off for users who ask for it. The fake telemetry readouts (`2.48×`, `8/23`, `DRAW · 23:47`) are removed — no placeholder data in UI is a project invariant.
- **Signin respects `callbackUrl`; magic-link redirects to the check-email page.** The signin action validates that `callbackUrl` is a safe relative path (starts with `/`, no `//`) and redirects there post-auth. The magic-link branch redirects to `/signin/check-email?email=...` instead of returning an inline `info` string — the orphaned page finally has a caller.
- **Atmosphere restored.** The conic-gradient mesh from the design canon lands as a composed `body::before` with screen-blended radial gradients plus the existing grain texture. Under `prefers-reduced-motion: reduce`, the mesh flattens to a static linear gradient.
- **Fonts load via `<link>`, not CSS `@import`.** The blocking `@import url('https://api.fontshare.com/...')` moves to a `<link rel="stylesheet">` in the root layout with `preconnect` to both Fontshare origins. True `next/font/local` self-hosting is still the destination — it requires committing the woff2 files, which is tracked as a follow-up.

## Consequences

- **Positive.** Adding an authed route is `src/app/(app)/<slug>/page.tsx` — no bespoke auth code, no shell wiring. Marketing gets a real landing-page feel (no disabled sidebar). Flipping `FLAG_CRASH=on` in one env var makes the game live across lobby, sidebar, and all downstream checks without a code change. Coming-soon cards no longer lie about being clickable. The game-art split is now <150 lines per file, under the 800-line budget, with motion respecting user preference.
- **Negative.** Two shells means two places to maintain the top-of-page treatment (brand lockup, sign-in CTA position). Acceptable because the surfaces diverge further as the app grows (authed TopBar carries balance + avatar; marketing header carries in-page anchor nav). The `(app)` layout does one DB read per navigation for the balance — unchanged from ADR-0009, still acceptable until the Phase 12 realtime channel lands.
- **Reversible.** Merging the two shells back would be mechanical. Moving to self-hosted fonts is additive (drop woff2 into `src/app/fonts/`, swap `<link>` for `next/font/local` calls, no downstream code changes). The nav-model split is the least reversible decision because several components now depend on `resolveGames()` / `userMeta` shapes — but that's the decision we want to be sticky.
